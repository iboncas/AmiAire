# TEST.md

## Objective

Run a short functionality/usability test of the web app with **5 convenience users**, using a **moderated qualitative usability test**. The goal is to detect the main usability issues in the app's key flows.

## Why 5 people

With 5 users, most major interface problems usually appear without needing a large sample. It is a fast, affordable, and sufficient method for an initial functionality validation.

## Features to test

For this app, the main features to test are:

1. Entering experiment dates.
2. Searching for and confirming a location.
3. Uploading a sensor image and analyzing it.
4. Confirming and saving the result.
5. Exploring the map, filtering sensors, and viewing details.

## Participants

Select 5 people who are reasonably close to the target user profile, for example:

- students,
- teachers,
- users with basic digital skills.

Avoid using only people who already know the app very well or who participated in its development.

## Preparation

Before starting:

1. Make sure the web app is working.
2. Prepare one valid sensor image.
3. Have a note-taking sheet ready.
4. If possible, record screen and audio with the participant's permission.

## How to run the test

Run sessions of **15-20 minutes** per participant.

### Instructions for the participant

Say something like:

> We are going to test the web app. We are not evaluating you, we are evaluating the application. Please complete the tasks as you normally would and think out loud while you navigate.

### Rules for you as moderator

- Do not explain how to do the task.
- Do not point to where they should click.
- Only intervene if the person is completely stuck.
- Write down doubts, errors, blockers, and spontaneous comments.

## Recommended tasks

### Task 1. Register an experiment

Scenario:

> You have carried out an experiment with a sensor and want to register it in the web app.

What the user should do:

- enter a start date,
- enter an end date,
- move to the next step.

### Task 2. Set the location

Scenario:

> You want to record the exact location of the experiment.

What the user should do:

- search for a location,
- review the map,
- confirm the position.

### Task 3. Upload the sensor image

Scenario:

> You have a photo of the sensor and want to analyze it.

What the user should do:

- upload the image,
- start the analysis,
- understand whether the image is valid or not.

### Task 4. Confirm and save the result

Scenario:

> The system has already detected the sensor area and you want to save the result.

What the user should do:

- review the detected area,
- save the result.

### Task 5. Explore the map

Scenario:

> You want to check sensors near a city and review their data.

What the user should do:

- open the map,
- filter by location or radius,
- enable or disable sensor types,
- select a sensor and read its details.

## What you should observe

For each task, note:

- whether they complete it or not,
- whether they need help,
- where they hesitate,
- what mistakes they make,
- what parts they do not understand.

## Problem classification

After the 5 sessions, classify the problems like this:

- **Critical**: prevents task completion.
- **Medium**: task can be completed, but with confusion or frustration.
- **Low**: minor issue that does not block the task.

At the end, you should have:

1. The problems that repeat most often across the 5 users.
2. A prioritized list of critical, medium, and low issues.
3. Concrete improvement proposals for the next iteration.
